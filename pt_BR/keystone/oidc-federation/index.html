      O que é Federação?

      A Federação Keystone permite que identidades de um Provedor de Identidade (IDP) sejam usadas em um provedor de Serviços (SP). Os diferentes protocolos e formatos suportados são OpenIDConnect (OIDC) e Security Assertion Markup Language (SAML).

      O IDP armazena e gerencia as credenciais do usuário e envia reclamações ou asserções ao SP. Isso permite que o cliente use suas credenciais corporativas para autenticar para um Bluemix Private Cloud sem enviar sua senha para seu Bluemix Private Cloud. O Keystone geralmente é sempre o SP, uma vez que ele consome identidades de recursos externos.

      O que é OpenID Connect (OIDC)?

      OpenID é um protocolo para autenticação. OAuth é um protocolo para autorização. OpenID Connect realiza ambas as ações.

      O OpenID Connect é uma camada de identidade simples na parte superior do protocolo OAuth 2.0. Ele permite que os Clientes verifiquem a identidade de um Usuário Final, com base na autenticação executada por um Servidor de Autorização. Ele também permite que os Clientes obtenham informações básicas do perfil sobre o Usuário Final, de uma maneira interoperável e semelhante ao REST.

      Este documento descreve um mecanismo para uma parte dependente do OpenID Connect para descobrir o Provedor OpenID do Usuário Final, e para obter informações necessárias para interagir com ele, incluindo seus locais de terminal do OAuth 2.0.

      Como funciona o Protocolo OpenID?

      1. A parte dependente envia a solicitação para o provedor OpenID para autenticar o Usuário Final.
      2. O provedor OpenID autentica o usuário.
      3. O provedor OpenID envia o token de ID e o token de acesso para a parte dependente.
      4. A parte dependente envia uma solicitação para o terminal de informações do usuário com o token de acesso recebido do provedor OpenID.
      5. O terminal de informações do usuário retorna as reclamações.

      Configurando o OIDC

      Um operador do Blue Box precisará trabalhar com um engenheiro de integração do IDP da empresa do cliente (por exemplo, Company XYZ) para configurar o OIDC.

      O operador do Bluemix Private Cloud precisará das seguintes informações:

      • URL autorizada de redirecionamento/retorno de chamada
      • ID do cliente e segredo do cliente
      • Reclamação do usuário - Reclamação do OIDC
      • A URL conhecida/de configuração que fornece todas as informações necessárias para acesso ao provedor.
      • Mapeamento padrão/inicial - A documentação sobre mapeamentos pode ser encontrada aqui: http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combination

      O operador do Bluemix Private Cloud deve trabalhar em conjunto com o engenheiro de integração para criar o mapeamento inicial/padrão. O mapeamento fornece o modo como os usuários remotos no IDP são mapeados para os grupos locais no SP. O restante do documento pode ajudar a esclarecer o que é um mapeamento e como ele é usado.

      Quando o operador do Bluemix Private Cloud termina de fazer a integração entre o IDP Company XYZ e o Keystone, um mapeamento chamado mapping-for-company-xyz-idp deve ser criado (ou um nome de mapeamento semelhante, contendo o nome do IDP).

      As etapas para o gerenciamento do mapeamento são:

      1. Gerenciar grupos (Usuários federados terão os mesmos direitos de acesso que o grupo)

      2. Gerenciar os mapeamentos

      Gerenciando grupos

      A Federação do Keystone trabalha federando usuários do Provedor de identidade para um grupo no Provedor de serviços. O acesso ao grupo é concedido designando uma função a esse grupo em um projeto. As funções disponíveis são as seguintes:

      • project_admin - A função admin para o projeto.
      • cloud_admin - A função admin para a instância da nuvem.
      • _member_ - Um membro de um projeto.
      • heat_stack_owner - Um proprietário para as pilhas de Heat.

      Para listar grupos:

      $ openstack group list

      Para criar um grupo:

      $ openstack group create --help
$ openstack group create <new group name here>

      Para listar funções:

      $ openstack role list

      Para listar designações de função:

      $ openstack role assignment list --name

      Para listar projetos:

      $ openstack project list

      Para obter ajuda ao consultar a inclusão da designação de função:

      $ openstack role add --help

      Para incluir uma designação de função para um grupo para ter uma função em um projeto:

      $ openstack role  add <role> --project <project> --group <group>

      Os usuários federados são mapeados em grupos no Provedor de serviços. Esta é a ordem das operações para conceder acesso para os grupos:

      1. Primeiro, crie ou edite o grupo.

      2. Em seguida, crie ou edite as designações de função para grupos nos projetos.

      Exemplo 1. Criando um grupo company_xyz_admin_group.

      $ source cloud_adminrc # Your openstack credentials file
$ openstack group list
$ openstack group create company_xyz_admin_group --or-show
$ openstack role assignment list --name --group company_xyz_admin_group

      Em seguida, crie a designação de função, se não existir.

      Estes comandos incluem a função cloud_admin para o grupo cloud_admin do projeto “demo”.

      $ openstack role add cloud_admin --group company_xyz_admin_group --project demo
$ openstack role assignment list --name --group company_xyz_admin_group

      Depois de executar o comando openstack role assignment list, você deve ver uma saída semelhante a esta:

      +-------------+------+---------------------------------+--------------+--------+-----------+
| Role        | User | Group                           | Project      | Domain | Inherited |
+-------------+------+---------------------------------+--------------+--------+-----------+
| cloud_admin |      | company_xyz_admin_group@Default | demo@Default |        | False     |
+-------------+------+---------------------------------+--------------+--------+-----------+

      Exemplo 2. Criando um grupo com acesso de membro.

      $ source cloud_adminrc #Your openstack credentials file
$ openstack group list
$ openstack group create company_xyz_member_group --or-show
$ openstack role assignment list --name --group company_xyz_member_group
$ openstack role add _member_ --group company_xyz_member_group --project demo
$ openstack role assignment list --name --group company_xyz_member_group

      Depois de inserir o comando openstack role assignment list, você deve ver uma saída semelhante a esta:

      +----------+------+----------------------------------+--------------+--------+-----------+
| Role     | User | Group                            | Project      | Domain | Inherited |

      Gerenciando mapeamentos

      Abaixo, estamos usando o exemplo do IDP Company XYZ e supondo que já exista um mapeamento para ele. Um usuário com acesso cloud_admin (usuário ou função) poderá editar o mapeamento. O usuário não poderá criar ou excluir mapeamentos.

      No Horizon, os mapeamentos podem ser gerenciados em Painel de identidade -> Federação -> Mapeamentos -> Editar

      O comando openstack mapping list mostra uma lista de mapeamentos. Ele deve mostrar o mapeamento mapping-for-company-xyz-idp.

      O comando openstack mapping show mapping-for-company-xyz-idp -f json mostra os detalhes do mapeamento fornecido. Ele deve conter uma listagem json das regras para o mapeamento.

      O comando openstack mapping set mapping-for-company-xyz-idp --rules newmappings.json atualiza e configura o mapeamento.

      O arquivo de mapeamento deve seguir este formato:

      [
   {
      "local": [
        <group>
      ],
      "remote": [
        <condition>
      ]
   }
]

      O arquivo de regras deve conter uma lista de regras que mapeiam os usuários remotos para um grupo local. A seção local contém uma regra que especifica o grupo para o qual os usuários remotos devem ser mapeados. O grupo local é necessário.

      {
  "group": {
    "domain": {
      "name": "Default"
    },
          "name": "company_xyz_admin_group"
        }
}

      A seção remota inclui regras que especificam para quais usuários remotos o acesso deve ser permitido.

      {
    "type": "group",
    "any_one_of": [
        "admin_group"
        ]
}

      Os tipos de atributos do usuário possíveis que podem ser fornecidos nas regras remotas são estes:

      As ações possíveis para a filtragem de atributos remotos são:

      empty
any_ony_of
not_any_of
blacklist
whitelist

      Para configurar o nome do usuário, é possível usar {0} na seção local para indicar o uso de um campo da seção remota. Em seguida, podemos inserir informações adicionais no novo nome de usuário Federado. É possível fazer referência à seção remota na seção local por meio do índice (“{i}”, em que i é o índice do campo remoto):

      [
  {
    "local": [
      {
        "user": {
          "name": "CompanyXYZ/{0}"
        }
      },
               ...
    ],
    "remote": [
      {
        "type": "HTTP_OIDC_CLAIM_EMAIL"
      },
               ...
    ]
  }
]

      Exemplo 4. Usando a API para o fluxo de códigos de autorização

      Este arquivo é oidc-authflow.py

      from keystoneauth1.identity.v3 import oidc
from keystoneauth1 import session
from requests_oauthlib import OAuth2Session
from keystoneauth1.identity.v3 import Token
from novaclient import client
import os

def main():

    auth_url = os.environ.get('OS_AUTH_URL')
    identity_provider = os.environ.get('OS_IDENTITY_PROVIDER')
    protocol = 'oidc'
    redirect_uri = os.environ.get('OS_REDIRECT_URI')
    scope = ['openid','profile','email']
    client_id = os.environ.get('OS_CLIENT_ID')
    client_secret = os.environ.get('OS_CLIENT_SECRET')
    discovery_endpoint = os.environ.get('OS_DISCOVERY_ENDPOINT')
    access_token_endpoint = os.environ.get('OS_TOKEN_ENDPOINT')
    authorization_endpoint = os.environ.get('OS_AUTHORIZATION_ENDPOINT')
    project_id = os.environ.get('OS_PROJECT_ID')

    oauth = OAuth2Session(client_id, redirect_uri=redirect_uri, scope=scope)
    authorization_url, state = oauth.authorization_url(authorization_endpoint,
                                                       access_type='offline',
                                                       approval_prompt='force')
    print( 'Please go to %s and authorize access' % authorization_url)
    authorization_code = raw_input('Enter the authorization code here: ')

    s = session.Session(verify=False)

    oidc_plugin = oidc.OidcAuthorizationCode(auth_url, identity_provider,
                                             protocol, client_id=client_id,
                                             client_secret=client_secret,
                                             access_token_endpoint=access_token_endpoint,
                                             discovery_endpoint=discovery_endpoint,
                                             access_token_type='access_token',
                                             redirect_uri=redirect_uri,
                                             code=authorization_code)

    unscoped_auth_token = oidc_plugin.get_unscoped_auth_ref(s)
    scoped_token = Token(auth_url=auth_url,
                         token=unscoped_auth_token._auth_token,
                         project_id=project-id,
                         reauthenticate=False)
    federated_session = session.Session(auth=scoped_token)
    nova = client.Client(version=2, session=federated_session)
    for server in nova.servers.list():
        print server.name

if __name__ == '__main__':
    main()

      Usando a API para o fluxo de senha

      Este arquivo é oidc-passflow.py

      from keystoneauth1 import session
from keystoneauth1.identity.v3 import oidc
from keystoneauth1.identity.v3 import Token
from keystoneclient.v3.client import Client

from novaclient import client import os


auth_url = os.environ.get('OS_AUTH_URL')
identity_provider = os.environ.get('OS_IDENTITY_PROVIDER')
protocol = 'oidc'
redirect_uri = os.environ.get('OS_REDIRECT_URI')
scope = ['openid','profile','email']
client_id = os.environ.get('OS_CLIENT_ID')
client_secret = os.environ.get('OS_CLIENT_SECRET')
discovery_endpoint = os.environ.get('OS_DISCOVERY_ENDPOINT')
access_token_endpoint = os.environ.get('OS_TOKEN_ENDPOINT')
authorization_endpoint = os.environ.get('OS_AUTHORIZATION_ENDPOINT')
project_id = os.environ.get('OS_PROJECT_ID')

def get_project_and_federated_session():
    s = session.Session(verify=False)

    # get an access token
    payload = {'client_id': client_id, 'grant_type': 'password',
                'username': '<username>',
                'password': '<password>', 'scope': 'openid',
                'connection':'Username-Password-Authentication'}

    response = s.post('https://<FQDN-OP>/oauth/ro',
                            data=payload,
                            authenticated=False)
    print(response.json())
    access_token = response.json()['access_token']

    oidc_plugin = oidc.OidcAccessToken(auth_url, identity_provider, protocol, access_token)
    response = oidc_plugin.get_unscoped_auth_ref(s)
    scoped_token = Token(auth_url=auth_url,
                         token=response._auth_token,
                         project_id=project_id,
                         reauthenticate=False)
    federated_session = session.Session(auth=scoped_token)
    return  federated_session


def main():
    scoped_session = get_project_and_federated_session()
    nova = client.Client(version=2, session=scoped_session)
    for server in nova.servers.list():
        print server.name

if __name__ == '__main__':
    main()

      3. Obtenha um código de autorização do seu Gerenciador de Identidade.

      4. O comando a seguir busca um token do Keystone que pode ser usado para executar qualquer comando até ele expirar:

          openstack --os-auth-type v3oidcauthcode --os-code <auth-code> --os-project-name <project-name> --os-project-domain-name Default token issue

      Exemplo:

      root@allinone-multiple:~# openstack --os-auth-type v3oidcauthcode --os-code 6LxfzO7GvCjmmcRa --os-project-name admin --os-project-domain-name Default token issue

      Notas adicionais

      Atualmente, existe um problema com usuários federados e com o recurso fideicomitente/fiduciário no Keystone, o que impede o usuário federado do Heat de delegar sua função heat_stack_owner.

      Será necessário usar uma solução alternativa para a delegação de confiança, designando a função heat_stack_owner diretamente para o usuário. Este é o usuário criado durante o login federado. O domínio de usuário é federado e pode ter o valor Nenhum (Federation=None).

      $ openstack user list

      O usuário federado deve ter um domínio de Nenhum: openstack user show *federated user*

      Aqui, o ID do usuário é necessário:

      $ openstack role add heat_stack_owner --user *federated user ID* --project *project*

      Outro problema conhecido é que um usuário federado tem a opção de mudar sua senha na página de configurações. Ele não poderá mudar a senha.
