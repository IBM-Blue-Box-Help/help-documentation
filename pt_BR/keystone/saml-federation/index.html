      O que é Federação SAML

      Security Assertion Markup Language (SAML) é um protocolo que permite que um usuário use a Conexão Única (SSO) com um Provedor de Identidade (IDP) e um Provedor de Serviços (SP). Quando um usuário efetua login, ele dá suas credenciais para o IDP. O IDP verifica as credenciais e uma Asserção SAML com os atributos do usuário é criada. Essa asserção é, então, passada para o SP e o SP verifica se ela foi enviada pelo IDP. Como o SP confia nas asserções do IDP, o usuário nunca tem que enviar suas credenciais para o SP.

      O Keystone agirá como o SP e autenticará usuários recebendo asserções de um IDP. Nessa documentação, vamos usar um IDP SAML fictício chamado IDP Company XYZ.

      Observe que quaisquer comandos administrativos nesta página requerem que o usuário tenha acesso ao cloud_admin.

      Integrando o IDP e o Keystone

      Um operador do Bluemix Private Cloud precisará trabalhar com um engenheiro de integração do IDP Company XYZ para configurar ambos os sistemas para que funcionem juntos.

      Um operador do Bluemix Private Cloud precisará do seguinte:

      • Terminal de URL de Metadados do IDP (Nós atualmente suportamos apenas recuperar Metadados por meio de uma URL e não um arquivo)
      • ID de entidade do IDP dos Metadados do IDP
      • Atributos do usuário - Esses são os atributos do usuário SAML passados pelo IDP. Eles podem ser encontrados na documentação do IDP ou nos logs do lado do servidor do IDP.
      • Mapeamento padrão/inicial - A documentação sobre mapeamentos pode ser encontrada aqui: http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combination

      O engenheiro de integração do IDP Company XYZ pode precisar do seguinte do Operador do Bluemix Private Cloud:

      • URL de metadados do provedor de serviços
      • URL de redirecionamento do provedor de serviços (Também conhecida como a URL de retorno de chamada, este é um valor incluído na lista de desbloqueio para o qual o IDP permitirá que o redirecionamento seja feito)

      O operador do Bluemix Private Cloud deve trabalhar em conjunto com o engenheiro de integração para criar o mapeamento inicial/padrão. O mapeamento é como os usuários remotos no IDP são mapeados para grupos locais no SP. O restante do documento pode ajudar a esclarecer o que é um mapeamento e como ele é usado. O engenheiro de integração também precisa do acesso cloud_admin. Este pode ser o usuário cloud_admin ou um usuário com a função do mesmo nome.

      Os atributos do usuário devem receber o nome de mapeamento. Por exemplo, os atributos podem ser fornecidos como:

      atributos:
- name: http://schemas.xmlsoap.org/ws/2005/05/identity/claims/emailaddress
  id: email
- name: http://schemas.xmlsoap.org/claims/Group
  id: group

      O nome dado é o que ele é dentro da asserção SAML. O ID é o que será usado nos mapeamentos do Keystone.

      Quando o operador do Bluemix Private Cloud termina de fazer a integração entre Company XYZ IDP e o Keystone, um mapeamento chamado mapping-for-company-xyz-idp deve ser criado (ou um nome de mapeamento semelhante, contendo o nome do IDP).

      As etapas para o gerenciamento do mapeamento são:

      1. Gerenciar grupos (Usuários federados terão os mesmos direitos de acesso que o grupo 2. Gerenciar os mapeamentos

      Gerenciando grupos

      A Federação do Keystone trabalha federando usuários do Provedor de identidade para um grupo no Provedor de serviços. O acesso ao grupo é concedido designando uma função a esse grupo em um projeto. As funções disponíveis são as seguintes:

      • project_admin - A função admin para o projeto.
      • cloud_admin - A função admin para a instância da nuvem.
      • _member_ - Um membro de um projeto.
      • heat_stack_owner - Um proprietário para as pilhas de Heat.

      Para listar grupos:

      $ openstack group list

      Para criar um grupo:

      $ openstack group create --help
$ openstack group create <new group name here>

      Para listar funções:

      $ openstack role list

      Para listar designações de função:

      $ openstack role assignment list --name

      Para listar projetos:

      $ openstack project list

      Para obter ajuda ao consultar a inclusão da designação de função:

      $ openstack role add --help

      Para incluir uma designação de função para um grupo para ter uma função em um projeto:

      $ openstack role  add <role> --project <project> --group <group>

      Os usuários federados são mapeados em grupos no Provedor de serviços.

      Esta é a ordem das operações para conceder acesso para os grupos:

      1. Primeiro, crie ou edite o grupo. 2. Em seguida, crie ou edite as designações de função para grupos nos projetos.

      Exemplo 1. Criando um grupo company_xyz_admin_group.

      $ source cloud_adminrc # Your openstack credentials file
$ openstack group list
$ openstack group create company_xyz_admin_group --or-show
$ openstack role assignment list --name --group company_xyz_admin_group

      Em seguida, crie a designação de função, se não existir. Estes comandos incluem a função cloud_admin para o grupo cloud_admin do projeto “demo”.

      $ openstack role add cloud_admin --group company_xyz_admin_group --project demo
$ openstack role assignment list --name --group company_xyz_admin_group

      Depois de executar o comando openstack role assignment list você deve ver uma saída semelhante a esta:

      +-------------+------+---------------------------------+--------------+--------+-----------+
| Role        | User | Group                           | Project      | Domain | Inherited |
+-------------+------+---------------------------------+--------------+--------+-----------+
| cloud_admin |      | company_xyz_admin_group@Default | demo@Default |        | False     |
+-------------+------+---------------------------------+--------------+--------+-----------+

      Exemplo 2. Criando um grupo com acesso de membro.

      $ source cloud_adminrc #Your openstack credentials file
$ openstack group list
$ openstack group create company_xyz_member_group --or-show
$ openstack role assignment list --name --group company_xyz_member_group
$ openstack role add _member_ --group company_xyz_member_group --project demo
$ openstack role assignment list --name --group company_xyz_member_group

      Depois de inserir o comando openstack role assignment list você deve ver uma saída semelhante a esta:

      +----------+------+----------------------------------+--------------+--------+-----------+
| Role     | User | Group                            | Project      | Domain | Inherited |
+----------+------+----------------------------------+--------------+--------+-----------+
| _member_ |      | company_xyz_member_group@Default | demo@Default |        | False     |
+----------+------+----------------------------------+--------------+--------+-----------+

      Gerenciando mapeamentos

      Abaixo, estamos usando o IDP Company XYZ e um mapeamento para ele já existe. Um usuário com acesso cloud_admin (usuário ou função) poderá editar o mapeamento. O usuário não poderá criar ou excluir mapeamentos.

      No Horizon, os mapeamentos podem ser gerenciados em Painel de identidade -> Federação -> Mapeamentos -> Editar

      O comando openstack mapping list mostra uma lista de mapeamentos. Ele deve mostrar o mapeamento mapping-for-company-xyz-idp.

      O comando openstack mapping show mapping-for-company-xyz-idp -f json mostra os detalhes do mapeamento fornecido. Ele deve conter uma listagem json das regras para o mapeamento.

      O comando openstack mapping set mapping-for-company-xyz-idp --rules newmappings.json atualiza e configura o mapeamento.

      O arquivo de mapeamento deve seguir este formato:

      [
   {
      "local": [
        <group>
      ],
      "remote": [       <condition>
      ]
   }
]

      O arquivo de regras deve conter uma lista de regras que mapeiam os usuários remotos para um grupo local. A seção local contém uma regra que especifica o grupo para o qual os usuários remotos devem ser mapeados. O grupo local é necessário.

      {
  "group": {
    "domain": {
      "name": "Default"
    },
          "name": "company_xyz_admin_group"
        }
}

      A seção remota inclui regras que especificam para quais usuários remotos o acesso deve ser permitido.

      {
    "type": "group",
    "any_one_of": [
        "admin_group"
        ]
}

      Os tipos de atributos do usuário possíveis que podem ser fornecidos nas regras remotas são estes:

      As ações possíveis para a filtragem de atributos remotos são:

      empty
any_ony_of
not_any_of
blacklist
whitelist

      Para configurar o nome do usuário, é possível usar {0} na seção local para indicar o uso de um campo da seção remota. Em seguida, é possível inserir informações para o novo nome de usuário federado. É possível fazer referência à seção remota na seção local por meio do índice (“{i}”, em que i é o índice do campo remoto):

      [
  {
    "local": [
      {
        "user": {
          "name": "CompanyXYZ/{0}"
        }
      },
               ...
    ],
    "remote": [
      {
        "type": "email"
      },
               ...
    ]
  }
]

      Usando o Horizon

      O painel do OpenStack no Provedor de identidade deve ter um menu suspenso com o rótulo Autenticar com SSO.

      • O menu suspenso deve mostrar uma etiqueta para o IDP Company XYZ. Selecione o IDP e clique em Conectar.
      • Em seguida, você será redirecionado para a página de login do IDP.
      • Depois de autenticar com êxito com o IDP, você deve ser redirecionado para a página do Horizon.

      Usando a CLI

      Nota: o cliente da Interface da Linha de Comandos (CLI) pode trabalhar apenas com IDPs que atendem a essas condições: 1. O IDP suporta o protocolo ECP SAML 2. O IDP suporta a autenticação básica em um GET

      A CLI oferece dois tipos de plug-ins: 1. v3unscopedsaml (Capturar um token sem escopo) 2. v3scopedsaml (Usar um token para executar ações com escopo definido)

      1. Exemplo sem escopo:

      Vamos criar um arquivo que podemos extrair em nosso ambiente para usar SAML:

      # This file is called companyxyz_saml_rc
export OS_IDENTITY_API_VERSION=3
export OS_PASSWORD=CompanyXYZPassword #replace this
export OS_AUTH_URL=https://<BB CLOUD FQDN>:5000/v3 #replace this
export OS_USERNAME=CompanyXYZUser #replace this
export OS_IDENTITY_PROVIDER=company-xyz-idp #replace this
export OS_PROTOCOL=saml2
export OS_AUTH_TYPE=v3unscopedsaml
export OS_IDENTITY_PROVIDER_URL=https://CompanyXYZ.example/idp/profile/SAML2/SOAP/ECP #replace this

      Depois de criar o arquivo e editar as opções, podemos executar o comando a seguir para obter um token sem escopo e registrar o ID:

      $ source companyxyz_saml_rc
$ openstack token issue

      2. Exemplo com escopo definido:

      Agora que temos um token sem escopo, podemos definir o escopo para um projeto e listar os servidores nesse projeto

      openstack server list --os-auth-type v3scopedsaml --os-token <os-token-id> --os-project-id <project-id>

      Usando as bibliotecas de API Python

      Nota: as bibliotecas de API Python podem funcionar apenas com IDPs que atendem a essas condições: 1. O IDP suporta o protocolo ECP SAML 2. O IDP suporta a autenticação básica em um GET

      O código a seguir usa um plug-in auth SAML para autenticar com o IDP, enviar a asserção SAML para o Keystone e recuperar um token sem escopo. O token sem escopo é, então, usado para obter um token com escopo definido do projeto. Em seguida, o script listará os servidores nesse projeto.

      Este arquivo é saml_list_servers.py:

      Será necessário extrair um arquivo antes de executar o script:

      # This file is called companyxyz_saml_python_rc
export OS_AUTH_URL=https://<BB CLOUD FQDN>:5000/v3 #replace this
export OS_USERNAME=CompanyXYZUser #replace this
export OS_PASSWORD=CompanyXYZPassword #replace this
export OS_PROJECT_NAME=demo
export OS_IDENTITY_PROVIDER=company-xyz-idp #replace this

      source companyxyz_saml_python_rc
python saml_list_servers.py

      Notas adicionais

      Atualmente, existe um problema com usuários federados e com o recurso fideicomitente/fiduciário no Keystone, o que impede o usuário federado do Heat de delegar sua função heat_stack_owner.

      Será necessário usar uma solução alternativa para a delegação de confiança, designando a função heat_stack_owner diretamente para o usuário. Este é o usuário criado durante o login federado. O domínio de usuário é federado e pode ter o valor Nenhum (Federation=None).

      $ openstack user list

      O usuário federado deve ter um domínio Nenhum: openstack user show <federated user>

      Aqui, o ID do usuário é necessário:

      $ openstack role add heat_stack_owner --user <federated user ID> --project <project>

      Outro problema conhecido é que um usuário federado tem a opção de mudar sua senha na página de configurações. Ele não poderá mudar a senha.
