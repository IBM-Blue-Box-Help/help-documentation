      O que é Federação K2K?

      A Federação Keystone to Keystone (K2K) é uma maneira de usar credenciais de uma instância do Keystone para efetuar login em várias instâncias do IBM Bluemix Private Cloud. Uma instância do Bluemix Private Cloud é designada como um Provedor de Identidade e outras instâncias da nuvem são designadas como Service Providers. O Provedor de Identidade armazena as credenciais do usuário e declara aos Service Providers que o usuário é, na verdade, um usuário válido. O ID e a senha do usuário nunca são enviados aos Service Providers, apenas as asserções do Provedor de Identidade são enviadas.

      Como funciona o K2K?

      O usuário se conecta ao Provedor de Identidade, recebe uma asserção SAML assinada do Provedor de Identidade e, em seguida, usa essa asserção SAML para autenticar nos Service Providers. O processo de login para manipular o fluxo SAML é manipulado pelo plug-in de autenticação de K2K (localizado em keystoneauth1). As credenciais são gerenciadas apenas no Provedor de Identidade e são aceitas pelos Service Providers por meio de asserções.

      Se várias nuvens do Bluemix Private tiverem sido configuradas com o recurso K2K, uma instância da nuvem será designada como o Provedor de Identidade e as outras serão designadas como Service Providers. Os Service Providers usam mapeamentos para selecionar os atributos do usuário do Provedor de Identidade que são utilizados para mapear para um grupo local. Depois que o usuário efetua login usando a federação K2K, o usuário recebe as funções designadas ao grupo.

      Por padrão, um mapeamento chamado mapping-for-k2k-federation é criado para as credenciais cloud_admin no Provedor de Identidade pelo qual elas são mapeadas para um grupo local chamado cloud_admin. Esse grupo tem o acesso cloud_admin da função para o projeto demo.

      O fluxo de configuração da federação K2K é o seguinte:

      1. Primeiro, gerencie grupos no Provedor de Serviços (Usuários federados terão os mesmos direitos de acesso que o grupo)

      2. Em seguida, gerencie mapeamentos no Provedor de Serviços (Mapeie usuários remotos do Provedor de Identidade para grupos locais do provedor de serviços)

      Gerenciando grupos

      A Federação K2K trabalha federando usuários do Provedor de identidade para grupos no Provedor de Serviços. O acesso ao grupo é concedido designando uma função a esse grupo em um projeto. As funções disponíveis são as seguintes:

      • project_admin - A função admin para o projeto.
      • cloud_admin - A função admin para a instância da nuvem.
      • _member_ - Um membro de um projeto.
      • heat_stack_owner - Um proprietário para as pilhas de Heat.

      Para usar os comandos de grupo, um arquivo de origem de credencial é requerido. Será necessário criar um arquivo contendo as credenciais e substituir as tags mostradas nos sinais de maior e menor < > com as informações apropriadas:

      #This file is cloud_adminrc
export OS_IDENTITY_API_VERSION=3
export OS_PASSWORD=<cloud_admin password> #Change this
export OS_AUTH_URL=https://<service provider fqdn>:5000/v3
export OS_USERNAME=cloud_admin
export OS_TENANT_NAME=demo
export OS_CACERT=/opt/stack/ssl/openstack.crt
export OS_NO_CACHE=True

      Depois que o arquivo de origem de credencial é feito, é possível extrair o arquivo e usar qualquer um dos comandos do OpenStack abaixo para gerenciar grupos e funções. Nota: esses comandos são executados no Provedor de Serviços.

      Extraia as credenciais no ambiente:

      $ source cloud_adminrc

      Para listar grupos:

      $ openstack group list

      Para criar um grupo:

      $ openstack group create --help
$ openstack group create <new group name here>

      Para listar funções:

      $ openstack role list

      Para listar designações de função:

      $ openstack role assignment list --name

      Para listar projetos:

      $ openstack project list

      Para obter ajuda ao consultar a inclusão da designação de função:

      $ openstack role add --help

      Para incluir uma designação de função para um grupo para ter uma função em um projeto:

      $ openstack role  add <role> --project <project> --group <group>

      Os usuários federados são mapeados em grupos no Provedor de serviços. Por padrão, um grupo chamado cloud_admin existirá se a federação K2K for ativada. Esse grupo tem a função cloud_admin no projeto “demo”.

      Esta é a ordem das operações para conceder acesso para os grupos:

      1. Primeiro, crie ou edite o grupo.
      2. Em seguida, crie ou edite as designações de função para grupos nos projetos.

      Exemplo 1. Criando um grupo cloud_admin.

      Os comandos abaixo mostram como gerar um arquivo com as informações de autenticação para o usuário cloud_admin do Provedor de serviços.

      $ source cloud_adminrc
$ openstack group list
$ openstack group create cloud_admin --or-show
$ openstack role assignment list --name --group cloud_admin

      Em seguida, crie a designação de função, se não existir. Estes comandos incluem a função cloud_admin para o grupo cloud_admin do projeto “demo”.

      $ openstack role add cloud_admin --group cloud_admin --project demo
$ openstack role assignment list --name --group cloud_admin

      Depois de executar o comando openstack role assignment list você deve ver uma saída semelhante a esta:

      # +-------------+------+---------------------+--------------+--------+-----------+
# | Role        | User | Group               | Project      | Domain | Inherited |
# +-------------+------+---------------------+--------------+--------+-----------+
# | cloud_admin |      | cloud_admin@Default | demo@Default |        | False     |
# +-------------+------+---------------------+--------------+--------+-----------+

      Exemplo 2. Criando um grupo com acesso de membro.

      Os comandos abaixo mostram como gerar um arquivo com as informações de autenticação para o usuário cloud_admin do Provedor de serviços.

      $ source cloud_adminrc
$ openstack group list
$ openstack group create demo_member_group --or-show
$ openstack role assignment list --name --group demo_member_group
$ openstack role add _member_ --group demo_member_group --project demo
$ openstack role assignment list --name --group demo_member_group

      Depois de inserir o comando openstack role assignment list você deve ver uma saída semelhante a esta:

      # +----------+------+---------------------------+--------------+--------+-----------+
# | Role     | User | Group                     | Project      | Domain | Inherited |
# +----------+------+---------------------------+--------------+--------+-----------+
# | _member_ |      | demo_member_group@Default | demo@Default |        | False     |
# +----------+------+---------------------------+--------------+--------+-----------+

      Gerenciando mapeamentos

      O usuário cloud_admin no Provedor de Serviços tem acesso para ler e configurar mapeamentos para a federação K2K. Observe que o usuário do Provedor de Serviços cloud_admin não pode excluir mapeamentos existentes ou criar mapeamentos novos.

      Para usar os comandos de mapeamento, um arquivo de origem de credenciais v3 é requerido. Você vai criar um arquivo contendo as credenciais e substituir as tags entre os sinais de maior e menor < > pelas informações apropriadas:

      Mais informações sobre como gerenciar mapeamentos podem ser encontradas na documentação do OpenStack: http://docs.openstack.org/developer/keystone/mapping_combinations.html http://docs.openstack.org/developer/keystone/federation/federated_identity.html#mapping-combinations

      Quando um usuário efetua logon no Provedor de Serviços por meio da federação K2K, usando sua conta do Provedor de Identidade, uma conta do usuário é criada localmente no Provedor de Serviços.Observe que a conta do usuário criada durante a federação é transitória. Ela pode ser excluída posteriormente sem aviso.

      No mapeamento padrão, o cloud_admin no usuário do Provedor de Identidade é federado, para que um usuário cloud_admin local seja criado no domínio federado. Esse usuário é diferente do usuário cloud_admin no domínio “Padrão”. O usuário cloud_admin transitório no domínio federado pode ser excluído no futuro.

      Nós precisamos do arquivo de origem criado a partir da seção anterior para que possamos executar os seguintes comandos de mapeamento:

      $ source cloud_adminrc
$ openstack mapping list
$ openstack mapping show mapping-for-k2k-federation
$ openstack mapping show mapping-for-k2k-federation --format json
$ openstack mapping show --help
$ openstack mapping set mapping-for-k2k-federation --rules <mappings file>

      O comando openstack mapping list mostra uma lista de mapeamentos. Ele deve mostrar o mapeamento mapping-for-k2k-federation.

      O comando openstack mapping show mapping-for-k2k-federation mostra os detalhes do mapeamento fornecido. Ele deve conter uma listagem json das regras para o mapeamento.

      O comando openstack mapping set mapping-for-k2k-federation --rules newmappings.json mostra como atualizar e configurar o mapeamento mapping-for-k2k-federation.

      O arquivo de mapeamento deve seguir este formato:

      [
   {
      "local": [
        <group>
      ],
      "remote": [       <condition>
      ]
   }
]

      O arquivo de regras deve conter uma lista de regras que mapeiam os usuários remotos para um grupo que existe no Provedor de Serviços. A seção local contém uma regra que especifica o grupo para o qual os usuários remotos devem ser mapeados. A regra de grupo local é necessária.

      {
  "group": {
    "domain": {
      "name": "Default"
    },
    "name": "cloud_admin"
  }
}

      A seção remota inclui regras que especificam para quais usuários remotos o acesso deve ser permitido.

      {
    "type": "openstack_user",
    "any_one_of": [
        "cloud_admin"]
}

      Os tipos de atributos do usuário possíveis que podem ser fornecidos nas regras remotas são estes:

      openstack_user
openstack_user_domain
openstack_roles
openstack_project
openstack_project_domain

      As ações possíveis para a filtragem de atributos remotos são:

      empty
any_ony_of
not_any_of
blacklist
whitelist

      Para configurar o nome do usuário, é possível usar {0} na seção local para indicar o uso de um campo da seção remota. Em seguida, é possível inserir informações para o novo nome de usuário federado. Para a federação K2K, é importante incluir algumas informações para indicar qual nuvem você está usando. É possível fazer referência à seção remota na seção local por meio do índice (“{i}”, em que i é o índice do campo remoto):

      [
  {
    "local": [
      {
        "user": {
          "name": "my_cloud/{0}"
        }
      },
               ...
    ],
    "remote": [
      {
        "type": "openstack_user"
      },
               ...
    ]
  }
]

      Utilizando o Horizon

      O painel do OpenStack no Provedor de identidade deve ter um menu suspenso com o rótulo Autenticar com Keystone para federação do Keystone.

      • Se você selecionar a opção Provedor de Identidade, a federação não será usada. O usuário será conectado à nuvem do OpenStack do Provedor de Identidade.
      • Se a opção Provedor de Serviços for selecionada, a federação será usada. A credencial do usuário do Provedor de Identidade será federada para o Provedor de Serviços.
      • Para alternar entre os provedores, o usuário precisará efetuar logout e selecionar novamente o provedor.

      Observe: a tela de login do Horizon para K2K para o IBM Bluemix Private Cloud 3.0.0 não é um recurso do OpenStack padrão. Ela pode mudar em liberações futuras.

      Usando as bibliotecas de API Python

      Em seguida, você fará um arquivo de origem que usará para autenticar usando o plug-in de autenticação K2K. O token OS_SERVICE_PROVIDER faz referência ao ID do Provedor de Serviços, que pode ser obtido executando-se o seguinte comando em relação ao Provedor de Identidade:

      $ openstack server provider list

      Aqui, podemos criar um arquivo de origem que será usado pelo script Python para listar as instâncias no Provedor de Serviços (Sp), usando as credenciais cloud_admin do Provedor de Identidade (IdP).

export OS_DOMAIN_NAME=Default
export OS_SERVICE_PROVIDER=<service provider id>

      É possível ver a lista de Service Providers executando os seguintes comandos no Provedor de Identidade:

      $ openstack service provider list

      O código a seguir usa o plug-in auth de Keystone-to-Keystone para obter um token sem escopo. Esse token sem escopo é usado para obter um token com escopo, que tem, então, o escopo definido para um dos projetos aos quais o usuário federado terá acesso. Depois de ter o token com escopo definido, você pode listar as instâncias do servidor no Provedor de Serviços.

      Crie um arquivo chamado list_servers.py:

      O script python pode ser executando fazendo o seguinte:

      $ source federationrc
$ chmod 700 list_servers.py
$ ./list_servers.py

$ openstack mapping set mapping-for-k2k-federation --rules federated_heat_stack_owner_mapping.json

      Será necessário usar uma solução alternativa para a delegação de confiança, designando a função heat_stack_owner diretamente para o usuário. Este é o usuário criado durante o login federado. O domínio de usuário é federado e pode ter o valor Nenhum (Federation=None).

      $ openstack user list

      O usuário federado deve ter um domínio Nenhum: openstack user show <federated user>

      Aqui, o ID do usuário é necessário:

      $ openstack role add heat_stack_owner --user <federated user ID> --project <project>

      Um problema conhecido é que um usuário federado tem a opção de mudar sua senha na página de configurações. Ele não poderá mudar a senha.
