      P. Como posso dizer quando minha máquina virtual foi reinicializada pela última vez?

      R. Você pode usar este comando:

      zgrep BOOT_IMAGE /var/log/kern.log*

      para ver quando o host (MV) foi inicializado pela última vez e, em seguida, usar esse registro de data e hora para consultar os logs para verificar se há alguma pista em relação aos problemas que você pode estar resolvendo.

      P. O que devo fazer se minha instância congelou e parou de responder?

      R. Recomendamos que você verifique o console da máquina virtual e veja se ele está completamente congelado ou se há alguma evidência de um pânico kernel ou outro erro. É possível obter a URL do console via:

      nova get-vnc-console VM-UUID novnc

      Observe o horário do console, se houver.

      Faça uma captura instantânea do console se ele estiver congelado e contiver informações úteis. Depois da reinicialização, se tiver ativado dumps de travamento na máquina virtual, você poderá verificar /var/crash para os arquivos vmcore (no RedHat) ou .crash (no Ubuntu) e analisar o dump a partir daí. Verifique também as estatísticas de atividade do sistema (sar) a respeito do horário do congelamento.

      Reinicialize a máquina se precisar que ela faça backup o mais rápido possível ou, melhor ainda, gire uma nova com base na mesma imagem. Se for necessário fazer uma investigação adicional na infraestrutura, diga-nos o horário do congelamento em UTC antes de reinicializar e o UUID da instância, e nós vamos ver se há alguma coisa a ser encontrada nos logs.

      Se precisar entrar no modo de usuário único de maneira confiável, por exemplo, para executar uma verificação do sistema de arquivos ou corrigir permissões de diretório SSH, nós fomos bem-sucedidos usando o console de tela inteira no navegador Safari. Será necessário pressionar o botão “enviar ctrl/alt/del” para reinicializar a MV enquanto estiver com login efetuado no console para poder capturá-lo na tela de carregamento GRUB. Quando estiver no menu de inicialização GRUB, pressione e para editá-lo. Para o modo de usuário único no CentOS, por exemplo:

      1. Role para baixo e apague a maior parte do final da linha (começando com o console) e deixe apenas console=tty lá.
      2. Depois de editá-la, a linha deve ser semelhante ao seguinte: kernel /boot/vmlinuz-2.6.32-431.el6.x86_64 ro root=UUID=8a057bba-2787-46a7-9436-fd6dc8a3aa85 rd_NO_LUKS rd_NO_LVM LANG=en_US.UTF-8 rd_NO_MD crashkernel=auto rd_NO_DM SYSFONT=latarcyrheb-sun16 KEYBOARDTYPE=pc KEYTABLE=us console=tty notsc single
      3. É possível também tentar incluir rw init=/bin/bash no lugar da parte ro da linha.
      4. Se o processo de inicialização parecer parar, pressione Enter algumas vezes.

      P. E se for preciso executar operações de resgate em minha instância?

      R.

      1. Obtenha um ISO de CD de resgate, por exemplo, de http://www.system-rescue-cd.org: 
        wget https://dronedata.dl.sourceforge.net/project/systemrescuecd/sysresccd-x86/4.9.6/systemrescuecd-x86-4.9.6.iso
      2. Instale isohybrid a partir do pacote sysutils: 
        apt-get install syslinux [ou yum install syslinux]
      3. Inclua o registro de inicialização principal no ISO para torná-lo um disco rígido inicializável 
        isohybrid systemrescuecd-x86-4.9.6.iso
      4. Faça upload do novo CD de resgate híbrido para o Glance: 
        openstack image create --file systemrescuecd-x86-4.9.6.iso --public sysrescuecd-4.9.6.iso
      5. Coloque a instância que precisa ser resgatada no modo de resgate: 
        nova rescue brokeninstance --image sysrescuecd-4.9.6.iso
      6. Efetue login na instância via console VNC no Horizon e execute qualquer operação de resgate que você precise executar. Pode ser necessário esperar de 1 a 2 minutos para que o CD de resgate se torne visualizável no console. Recarregue a tela do console depois de esperar.

      (/img/pt_BR/systemrescuefromvnc.png)

      A partir daqui é possível usar ferramentas, como:

      blkid
fsck /dev/vdb1
mount -t ntfs /dev/xxx /mnt/windows -o ro
ntfs-3g /dev/sda1 /mnt/windows
      1. Quando estiver satisfeito com a correção da instância, tire-a do modo de resgate e inicie-a a partir do disco de inicialização normal novamente usando o comando: 
        nova unrescue brokeninstance

      P. Por que não posso redimensionar minha instância de m1.tiny para m1.medium?

      É possível que a instância realmente tenha sido criada no momento em que o tipo m1.tiny tinha um disco efêmero maior. O processo de redimensionamento é verificar isso e o redimensionamento irá falhar se esse valor não for encontrado.

      Pode ser necessário abrir um chamado de suporte para obter ajuda para verificar se esta é a causa da falha. A equipe de suporte pode ver um erro nos logs, algo semelhante ao seguinte:

      Exceção durante a manipulação da mensagem: erro de redimensionamento: não é possível redimensionar o disco para menos.

      Como alternativa, algumas boas opções para avançar sem abrir um chamado seriam configurar o tipo m1.medium para o mesmo tamanho do disco efêmero maior do que era antes ou seguir a solução alternativa aqui para criar uma nova instância da captura instantânea: /bluebox/help-documentation/gettingstarted/commontech/Instance_Resize/

      P. O que causou a parada das portas em Neutron?

      A parada das portas pode ocorrer em Neutron ao instanciar as MVs usando o Heat. Acredita-se que este problema esteja relacionado a um defeito no OpenStack Neutron (liberação Mitaka) devido a uma condição de disputa em Neutron. Portanto, nós o estamos tratando como uma limitação conhecida. Esta condição também pode causar a falha de metadados juntamente com a parada das portas.

      Existem várias soluções alternativas:

      Solução alternativa nº 1. Se possível, as instâncias podem ser implementadas sem usar o Heat. A implementação manual parece contornar essa condição.

      Solução alternativa nº 2. Outra solução alternativa é introduzir um atraso no Heat, usando WaitCondition da API. No entanto, WaitCondition pode não ser ideal neste cenário.

      • WaitCondition funciona juntamente com WaitConditionHandler. Esse manipulador cria uma URL/URI que é monitorada para uma sequência de texto. A presença dessa sequência indica o sucesso de uma MV implementada e a ausência dessa sequência indica falha. Depois que uma MV é instanciada, ela pode ser instruída a preencher a URL/URI com a sequência apropriada de texto por meio de uma chamada API PUT.

      • Atualmente, não existe nenhum mecanismo em nossa implementação do OpenStack para um roteador L3 fazer essa chamada API. Nossa equipe de engenharia sugere usar ‘depends_on’. (Veja a opção 3, a seguir).

      Solução alternativa nº 3. A melhor solução alternativa é usar depends_on. Ela cria um cenário na orquestração para que a implementação de uma MV “dependa” da existência de outros recursos definíveis: por exemplo, uma Rede e Roteador L3.

      Um pequeno exemplo depends_on:

      add_router_interface:
   type: OS::Neutron::RouterInterface
   properties:
     router: { get_resource: router}
     subnet: { get_resource: subnet}


server1:
    type: OS::Nova::Server
    depends on: add_router_interface
    properties:
      image: { get_param: image }

      Mais detalhes sobre Dependências Explícitas estão disponíveis no link a seguir:

      • https://wiki.openstack.org/wiki/Heat/Blueprints/hot-software-config-spec

      P. Quais portas precisam ser abertas para o OpenStack?

      Será necessário abrir pelo menos as portas 67/68 para DHCP e a porta 80 para metadados. Além disso, se quiser que as MVs consigam acessar os serviços de API do OpenStack, inclua as portas documentadas aqui:

      /bluebox/help-documentation/gettingstarted/commontech/FAQ/#which-portsurls-etc-are-publicly-accessible-on-my-bluemix-private-cloud-and-for-what-purpose

      Se o conjunto mínimo requerido de portas não for aberto, você poderá ter problemas com cloud_init de modo que suas instâncias não consigam iniciar com êxito.

      P. O que significa este erro? “Não é possível recuperar informações de limite de volume.”

      Frequentemente, quando carrego uma página no painel do Horizon, eu vejo um menu pop-up vermelho que diz: “Erro: não é possível recuperar informações de limite de volume.” O que isto quer dizer?

      Esse erro é inofensivo, mas ocorre porque não há nenhum armazenamento de bloco Cinder na implementação.

      O erro pode ser resolvido removendo o terminal do Cinder do serviço de Keystone. Isso pode exigir uma atualização dos arquivos de configuração de automação e será necessário que a equipe de Suporte faça uma verificação. Abra um chamado de suporte se precisar de ajuda para resolver essa mensagem de erro.

      Este é um exemplo das etapas a serem seguidas pelo técnico de suporte:

      # openstack endpoint list | grep cinder 
| 19a90853883049ed92cf4af82a24c0db | RegionOne | cinderv2     | volumev2       | True    | name  | https://... | 
| 2c3e8d0a15304788a9dc1806917b61ee | RegionOne | cinderv2     | volumev2       | True    | other     | https://... | 
| 39434faf90484a208b61a517926042b2 | RegionOne | cinderv2     | volumev2       | True    | name2    | https://... | 

# openstack endpoint delete 19a90853883049ed92cf4af82a24c0db 
# openstack endpoint delete 2c3e8d0a15304788a9dc1806917b61ee 
# openstack endpoint delete 39434faf90484a208b61a517926042b2 

# openstack service list | grep cinder 
| ef7a982aaf224c62a09c6e813ac66141 | cinderv2   | volumev2       | 

# openstack service delete ef7a982aaf224c62a09c6e813ac66141

      Verifique se a automação está correta:

      $ grep -A2 ^cinder ../all.yml 
cinder:  
  enabled: False 
cinderv2:  
  enabled: False

      P. Por que o acesso à porta 25 não está funcionando para o meu e-mail de saída?

      R. O SoftLayer não permite mais conexões de saída por meio da porta TCP 25 (SMTP) em novas contas. Consulte o artigo a seguir para obter mais informações:

      https://knowledgelayer.softlayer.com/procedure/outbound-email-port-25
