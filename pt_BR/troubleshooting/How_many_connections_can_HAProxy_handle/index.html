      How Many Simultaneous Connections can HAProxy Handle?

      • by Lampros Chaidas ● October 21, 2016

      • Tags:

        • troubleshooting

        • haproxy

        • maxconn

        • load testing

      Observe: este artigo NÃO está especificamente relacionado ao OpenStack Neutron LBaaS, apenas ao HAProxy em execução dentro das instâncias.

      Recentemente, eu estava vendo a configuração de balanceamento de carga de um cliente. Ao examinar a página de estados do HAProxy com o cliente, a pergunta surgiu: “Quantas conexões simultâneas ele pode realmente suportar?”

      O número máximo de conexões por servidor de backend é realmente importante? Este artigo abrange uma maneira de determinar o número de conexões simultâneas que HAProxy pode manipular.

      Nós vamos instalar HAProxy em um CentOS 6 VPS e executar uma série de testes para encontrar a resposta. Em um VPS diferente, vamos configurar nosso utilitário comparativo.

      Requisitos:

      • 2x CentOS 6.x VPS

      As especificações para os VPSs usados nesses testes eram:

      • 2 GB de RAM
      • 2 núcleos
      • CentOS 6.7

      Instalando HAProxy no HAProxy VPS:

      Etapa 1. Instale o HAProxy IUS (Inline with Upstream Stable) repo para CentOS 6:

        yum -y install https://centos6.iuscommunity.org/ius-release.rpm

      Etapa 2. Em seguida, obtenha a versão mais recente do HAProxy - no momento da gravação é 1.6.9.

        yum -y install haproxy16u && chkconfig haproxy on

      Etapa 3. Mova sua configuração do HAproxy padrão para algum outro lugar, no caso de você precisar revisá-la mais tarde:

        mv /etc/haproxy/haproxy.cfg /etc/haproxy/haproxy.cfg-default

      Etapa 4. Configure rsyslog para receber logs do HAProxy:

        sed -i -- 's/#$ModLoad imudp/$ModLoad imudp/g' /etc/rsyslog.conf
  sed -i -- 's/#$UDPServerRun 514/$UDPServerRun 514/g' /etc/rsyslog.conf
  echo "\$UDPServerAddress 127.0.0.1" >> /etc/rsyslog.conf
  echo "local2.* -/var/log/haproxy.log" >> /etc/rsyslog.conf

      Os comandos anteriores configuraram rsyslog para atender à porta UDP 514 e limitá-la apenas às conexões localhost. A última linha configura os logs HAProxy para ir para /var/log/haproxy.log.

      Etapa 5. Reinicie rsyslog para que as mudanças entrem em vigor:

        service rsyslog restart

      Etapa 6. Inclua uma nova configuração do HAProxy - /etc/haproxy/haproxy.cfg - observe a linha maxconn 20. De acordo com a página do manual “Os proxies vão parar de aceitar conexões quando este limite for atingido”. Em outras palavras, esta configuração nos permite apenas até 20 conexões ao mesmo tempo.

      Qual é a linha fullconn 20? Na página do manual, ele está definido como “o número de conexões no backend que farão com que os servidores usem o número máximo de conexões.”

      global
daemon
# we need the socket for hatop statistics
stats socket /var/run/haproxy.sock mode 600 level admin
log /dev/log local4
maxconn 20

defaults
log global
timeout connect 4000
timeout client 42000
timeout server 43000

listen lampros-pool-1
bind *:80
mode http
balance roundrobin
fullconn 20
server go_http_server_1 127.0.0.1:8080 cookie go_http_server_1 check inter 2000 rise 2 fall 3

      Instalando a linguagem Go - Faça isso em ambos os servidores:

      Instale a linguagem Go juntamente com o utilitário screen e git. Nós precisamos dessa linguagem em ambos os servidores porque, no HAProxy VPS, vamos compilar um servidor de aplicativos muito básico. No VPS comparativo, precisamos de Go para podermos executar o utilitário comparativo gobench.

      yum -y install git screen nc
cd /tmp

      No servidor HAProxy, vamos configurar um servidor de aplicativos básico usando Go:

      Etapa 1. Crie um diretório chamado /root/slowserver

        mkdir -pv /root/slowserver

      Etapa 2. Dentro dele, inclua um arquivo chamado /root/slowserver/slowserver.go com o seguinte conteúdo:

      package main

import (
    "fmt"
    "net/http"
    "time"
)

func slow_handler(w http.ResponseWriter, r *http.Request) {
    time.Sleep(3000 * time.Millisecond)
    fmt.Fprintf(w, "I'm slooooow")
}

func fast_handler(w http.ResponseWriter, r *http.Request) {
    fmt.Fprintf(w, "I'm FAST!!!!")
}

func main() {
    http.HandleFunc("/fast", fast_handler)
    http.HandleFunc("/slow", slow_handler)
    http.ListenAndServe(":8080", nil)
}

      Etapa 3. Compile esse arquivo com go build:

        cd /root/slowserver
  go build

      Etapa 4. Inicie nosso slowserver em uma sessão de tela separada, para que fique no plano de fundo:

        screen -d -m -t go_slowserver /root/slowserver/slowserver

      Etapa 5. Confirme se ele está executando usando netstat -ntlp | grep slowserver - você deve ver a porta na qual ele está atendendo e também o ID do processo. Esta é a saída obtida:

        netstat -ntlp | grep slowserver
  tcp        0      0 :::8080                     :::*                        LISTEN      5204/slowserver

      Teste de carga - Determinando o número de conexões simultâneas que o HAProxy pode manipular

      Vamos ver quantas conexões simultâneas o HAProxy pode manipular. Antes de começarmos, não vamos esquecer um detalhe básico sobre o HAProxy: “Por padrão, o HAProxy opera no modo keep-alive com relação a conexões persistentes: para cada conexão, ele processa cada solicitação e resposta, e deixa a conexão inativa em ambos os lados entre o final de uma resposta e o início de uma nova solicitação.”

      Essa instrução significa que conexões de longo prazo terão um impacto maior sobre o servidor do que as de curto prazo. Dessa forma, vamos executar os testes em relação à função “slow” do nosso servidor de aplicativos.

      Conclusão: teste de carga - Determinar o número de conexões simultâneas que o HAProxy pode manipular

      A conclusão a que podemos chegar a partir do experimento anterior é que, para testar a capacidade (número máximo de clientes conectados ao HAProxy), precisamos usar um servidor que abra uma conexão TCP, mas a mantenha aberta por um período estendido de tempo (3 segundos). Isso é o oposto de testar a velocidade de um servidor – onde quanto maior o tráfego colocado nele, melhores resultados serão obtidos.

      A segunda conclusão é que suas conexões máximas (maxconn) precisam ser maiores que a taxa de solicitação (por exemplo, 100 clientes) * tempo (3 segundos) para atender a cada solicitação.

      Por exemplo:

      maxconn=100*3 => maxconn=300
